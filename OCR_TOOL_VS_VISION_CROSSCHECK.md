## 三、差异归类与根因

| 类型 | 出现频次 | 典型示例 | 根因 |
|------|----------|----------|------|
| 顶部艺术字/logo 切碎 | img6/7/10 共 3 次 | img10 `yro R` | OCR 把品牌艺术字当正文识别并切碎；视觉能识别为 logo |

| 行边界/表格错位 | img6 规格切开、img10 表头串行 | tbpu 阅读顺序在多列/小字表格上偶发错误 | tbpu GapTree + ParagraphParse 的已知边界 |

| brand/name 全空 | 5/5 | 规则抽取无中文品牌库 | LLM 未启用 |

## 四、核心结论

1. **整管线已落实并跑通**：5 张图全部经 `cli.py build` → `get_adapter(.jpg)` → `ImageTableAdapter.extract` → `tbpu` 排版 → `structure` 抽取 → NDJSON。Manifest 干净（ocr_pending=0、skipped_files=0）。

2. **OCR 是"字符级精确转录"基线，多模态视觉是"语义级理解"基线，两者互为补充**：
   - OCR 强项：成分表**数据行**（img8/10 完全一致）、许可证号、电话、条码
   - 视觉强项：彩色提示框/长说明/品牌 logo/配料表 等 OCR 易漏区域
   - **OCR 漏的"配料表""食用方法""贮存条件""彩色提示"恰好是营养品最关键字段**——需 LLM 层在结构化阶段基于"图+OCR 文本"补全

3. **结构化抽取瓶颈在 LLM 未启用**：当前 `llm.kind=none` → 只能 rule-only，5/5 张图 brand/name/manufacturer 全空；img6 出现 reg_number 越界、img7 出现 net_content 误抓。**生产环境应启用 `DATAPROC_LLM_KIND=openai/ollama`**，让 `structure()` 用 LLM 做字段抽取与边界判断。

4. **OCR 误差无损于"不编造"承诺**：所有 5 张都成功 OCR 进 corpus，**不保证字符全对**，但 LLM 在结构化阶段可基于"图+OCR 文本"做语义补全（**不**让 LLM 改写 OCR 事实）。

## 五、立即可推的优化

| 优先级 | 优化项 | 预期收益 |
|--------|--------|----------|
| P0 | 启用 LLM（`DATAPROC_LLM_KIND=openai/ollama`），让 `structure()` 走 LLM 抽取 | 5/5 张图的 brand/name/manufacturer 从"全空"到可填充；reg_number 越界/净含量误抓消失 |
| P0 | LLM 抽取 prompt 提供"图+OCR 文本"双输入，让 LLM 补 OCR 漏的提示框/配料/食用方法（**不**改写 OCR 事实） | img8 配料/食用方法、img9 红色提示、img7 顶部抗生素提示 等关键字段召回 |
| P1 | 给 reg_number/净含量抽取加严格边界正则（只匹配"国食健注 G\d+""SC\d+""净含量\d+g"之类） | 杜绝 reg_number 越界、净含量误抓 |
| P1 | 对艺术字 logo 区域做"跳过"启发式（位置/颜色），不当正文识别 | 减少 img6/7/10 开头碎片噪声 |
| P2 | CPU 推理 1~2 分钟/张偏慢，端侧建议评估 PP-OCRv6_mobile / INT8 量化 | 端侧可用性 |

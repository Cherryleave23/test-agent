3 · 端到端消息流（一条用户消息的生命周期）

👤 员工微信
📡 iLink API
🧭 Gateway
👶 Baby/Session
🤖 Agent+LLM
1
入站：ILinkClient getUpdates 长轮询拿到 IncomingMessage（message_id / from_user_id / content）。
ent=cfg.enterprise_id，emp=from_user_id，conv=from_user_id（DM：每员工一会话）
2
会话加锁 + 去重：session_key(ent,emp,conv) 取锁 → get_or_create；message_id 已见则跳过（幂等）。
3
约束处理（方向B+方向A）：
方向B：规则抽取本轮约束，确定性累积进会话状态（无 LLM）。
方向A：历史超阈 → LLM 压缩合并（合并而非替换，A3 修复）。
4
宝宝链路（若 baby_profile_enabled 且 baby_store 已装配）：
熔断检查：连续消歧失败 ≥ 阈值 → 降级「仅产品问答」。
resolve_and_archive：LLM 判归属 + 抽属性 + 混合建档安全网（自动建档 / 第三方不建）+ 主动归档。
设焦点宝宝 set_focus_baby；解析失败计数；焦点切换时刷新约束（A4/A5，消除旧宝宝残留）。
产出 baby_block（注入上下文）+ baby_profile_obj（检索查询融合）。
5
RAG 回答 Agent.answer 六步：
① _enrich_query：用 baby_profile 融合检索查询，提升命中。
② store.retrieve：embedding + rerank，top_k=5（按 enterprise_id 隔离）。
③ _build_context：拼检索片段。
④ _build_messages：稳定(企业system+约束) → 半稳定(baby_block) → 动态(【企业知识库】context)。
⑤ provider.complete(messages, cache_control)：LLM 生成。
⑥ 附引用 + 免责声明。
6
回写 + 出站：append_turn(user/assistant) → client.send_message 经 iLink 回员工微信（出站受 egress 白名单约束）。

5 · 安全加固（上线前必做 P0，已落地）

common/egress.EgressPolicy + AllowedAsyncClient 应用层出网域名强制。默认放通 iLink 两域名 + 显式 LLM 端点；AGENT_EGRESS_ENFORCE=1 开启拦截，AGENT_EGRESS_EXTRA_HOSTS 逃生阀。

6 · Prompt Caching（成本优化）

核心思想：让前缀稳定。 把「跨调用一致」的内容放在 messages[0]（system），每轮变量（焦点/历史/当前句）放在其后。
  • 消歧：稳定前缀 = 指令 + 已知宝宝清单（JSON，ORDER BY 保证序列化一致）→ 开 cache_control。
  • RAG：稳定企业 prompt → 半稳定 baby_block → 动态检索 context（顺序即缓存键）。
  • 效果：同员工会话前缀命中缓存，input token 费用降 50–90%；Anthropic 显式断点，OpenAI/兼容端点靠自动前缀缓存。
  • 监控：_report_cache_hit 解析 usage.prompt_tokens_details.cached_tokens 记命中率。

# 数据模型与数据库设计（端侧 + 厂商侧，三部分）

> 支撑 O1（Chroma，2026-07-20 由 SQLite-vec 改换）与 MOD-kb / MOD-ingest / MOD-deploy。本文记录**用户已确认的数据库结构事实**
> 与**已确认的数据来源/维护决策**；营养品 schema 由 agent 按用户委托提议（标注「提议」）。
> 用户原话要点：端侧数据库两部分（HQ 知识库共享 + B 端每企业不同）；B 端产品含奶粉（14 必填字段）/
> 营养品（类似）/尿不湿/服务；并补充了**总部商品库**复用思路与多种入库方式。

## 一、总览：数据 = 三部分

| 部分 | 名称 | 位置 | 范围 | 形态 |
|------|------|------|------|------|
| A | **HQ 知识库** | 端侧实例（随产品分发） | **跨企业共享、只读** | 非结构化 → 向量分块（育儿知识 + 营养成分解析） |
| B | **HQ 商品库** | **厂商侧（我们）** | 跨企业累积增长 | 结构化产品目录（奶粉/营养品/尿不湿…） |
| C | **B-end DB** | 端侧实例（每企业） | **每企业隔离** | 结构化产品目录（由 HQ 商品库播种 + 企业独有补充） |

> 区别：A 是「知识」，随镜像下发到每个实例、只读；B 是「厂商侧商品资产」，接入新企业时用来
> **复用/拼接**，不随实例下发（或仅作 onboarding 参考）；C 是「该企业实际部署、被问答检索」的库。

---

## 二、A. HQ 知识库（共享，随分发）

- **内容**：大量育儿知识 + 营养成分解析（由**厂商策划维护，随产品镜像分发**到每个实例）。
- **形态**：文档/文章 → 分块 → 嵌入 → 向量检索。
- **隔离**：跨企业共享、实例内只读；不参与企业隔离。
- **检索**：语义向量检索；与 B-end DB 联合排序（MOD-agent）。

---

## 三、B. HQ 商品库（厂商侧，累积复用）

- **内容**：我们（ToB 厂商）随接入企业增多而**累积的结构化产品目录**（奶粉/营养品/尿不湿…）。
- **作用（onboarding 加速）**：新企业接入时——

  ```
  查 HQ 商品库：
    商品已存在 ──▶ 直接拼接(splice)进该企业 B-end DB（省去重新解析）
    商品不存在 ──▶ 用 PDF/图片解析 / Excel导入 / 爬虫 获取，填入 schema，
                  并回填 HQ 商品库（供后续企业复用）
  ⇒ 该企业 B-end DB = HQ 复用部分 + 企业独有部分
  ```
- **形态**：与 B-end DB 同源的结构化产品表（奶粉 14 字段 / 营养品提议字段 / 灵活字段）。
- **位置**：厂商侧（不随单实例下发；onboarding 时读取并播种到目标企业实例）。

---

## 四、C. B-end DB（每企业隔离，结构化）

- **内容**：客户产品信息——**奶粉 / 营养品 / 尿不湿 / 独特服务**。
- **形态**：结构化产品目录，按品类 schema；检索 = 精确字段查询 + 语义检索（混合）。
- **数据来源（已确认，可多选）**：
  1. **PDF/图片解析抽取**：产品册 PDF / 图片 OCR → 抽取字段填入 schema（MOD-ingest）。
  2. **Excel/CSV 批量导入**：企业导入产品表。
  3. **爬虫**：官网商品页抓取（来源为企业自有/已授权）。
  4. **HQ 商品库复用拼接**：已在 HQ 的商品直接复用（见第三节）。
- **隔离**：每企业独立库/表（`enterprise_id` 隔离）。

## 五、对模块的影响

- **MOD-kb**：三类存储概念——
  - HQ 知识库（共享向量，随分发）、
  - HQ 商品库（厂商侧，播种用）、
  - B-end DB（每企业**结构化产品表 + 向量列**，混合检索）。
  - B-end 奶粉表示例：`products_milk(id, enterprise_id, name, brand, stage, age_range, price,
    origin, milk_origin, ptype, reg_number, manufacturer, ingredients, nutrition, highlights, embedding)`。
  - B-end 营养品表示例（schema 见 §4.2，已确认）：`products_nutrition(id, enterprise_id, name,
    brand, category, audience, dosage_form, age_range, price, origin, manufacturer, health_license,
    efficacy, ingredients, nutrition, highlights, cautions, embedding)`。
- **MOD-ingest**：数据源 = PDF/图片解析 + Excel 导入 + 爬虫 + **HQ 商品库复用拼接**；
  抽取/导入的字段填入品类 schema；新商品回填 HQ 商品库。
- **MOD-agent**：检索联合 **HQ 知识库 + B-end DB**，按企业上下文排序融合。
- **MOD-deploy**：分发 **HQ 知识库** 到每实例；onboarding 时从 **HQ 商品库** 播种该企业 B-end DB。

## 六、已确认决策（用户拍板）

- **Q-DB1 B 端数据来源**：PDF/图片解析 + Excel/CSV 导入 + 爬虫 + **HQ 商品库复用拼接**（多源并存）。
- **Q-DB3 HQ 知识库**：由**厂商策划维护、随产品分发**（共享只读）。
- **Q-DB4 尿不湿/服务**：**企业自定义灵活字段**（不强制固定 schema）。
- **Q-DB2 营养品 schema**：**已确认**（agent 提议，用户 2026-07-20 认可，见 §4.2）。

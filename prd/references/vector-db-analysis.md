# 向量库选型分析（支撑 O1 决策）

> 用途：为「企业定制知识库（MOD-kb）」选择向量存储。本文做**分析与权衡**。
> **决策（O1，用户已拍板）= Chroma（嵌入式 PersistentClient）**；**2026-07-20 由 SQLite-vec 改换**。
> 触发重评条件：极端边缘资源受限（仅能跑单文件库）→ 可回退 SQLite-vec；或需高并发多副本 → Qdrant。

## 〇、决策记录（O1）

- **选定：Chroma（嵌入式 PersistentClient）**。理由（用户 2026-07-20 要求）：原生 metadata 过滤天然强化
  `enterprise_id` 企业隔离、成熟混合检索、嵌入式 `PersistentClient` 契合「1 家 1 实例」端侧部署、规模扩展
  平滑（无需像 SQLite-vec 那样到百万级再换）。代价：比 SQLite-vec 重（依赖 onnxruntime/duckdb，持久化为目录）。
- **规模上限**：Chroma 百万~千万级向量顺畅；单实例单企业场景下远超所需。
- **隔离保证**：每实例独立 Chroma 持久化目录（物理隔离）+ 检索时 `enterprise_id` metadata 过滤（防御纵深）
  + SQLite FTS5 关键词召回与 Chroma 向量召回做 RRF 融合。

## 一、在这个项目里向量库到底干什么

母婴企业的知识（商品页、说明书 PDF、规格表、FAQ）经 `MOD-knowledge-ingest` 切成分块（chunk），
每块用**中文 embedding 模型**转成一串向量；向量库负责：

1. **写入**：存 `(向量, 原文, 元数据[来源/产品类目/采集时间])`。
2. **检索**：用户提问 → 转向量 → 近似最近邻（ANN）召回最相关 chunks。
3. **过滤**：按 `enterprise_id` / 产品类目等元数据圈定范围（**企业隔离的关键**）。
4. （可选）**混合检索**：向量召回 + 关键词（BM25）召回融合，提升「规格参数」类精确查询。

> 质量上，**embedding 模型的选择通常比向量库本身更影响效果**。向量库决定的是
> 规模、性能、运维与隔离能力，不是语义质量。

## 二、候选对比

| 选项 | 形态 | 运维负担（端侧） | 规模上限 | 元数据过滤 | 混合检索 | 隔离方式 |
|------|------|------------------|----------|------------|----------|----------|
| **Chroma** | 嵌入式（Python，可持久化到目录/SQLite） | **极低**，进程内即可 | 百万级向量顺畅；千万级开始吃力 | 支持（where 过滤） | 需自建/接 rank_bm25 | 每企业独立 collection |
| **Qdrant** | Rust 服务（也可 embedded/in-process 模式） | 低~中（服务态需起进程；embedded 接近零运维） | 千万~亿级 | **强**（payload 过滤、量化） | 内置 sparse+vsm 混合 | 每企业独立 collection + 过滤 |
| **pgvector** | Postgres 扩展 | 中（若已用 PG 则零额外） | 百万~千万（依赖 PG 调优） | 借 SQL WHERE | 借 SQL 全文 | 借表/行级隔离 |
| **Milvus** | 分布式向量库 | **高**（组件多，不适合轻量端侧） | 十亿级 | 强 | 内置 | collection 隔离 |
| **FAISS** | 纯库（in-memory，非服务） | 低（但需自己管持久化/服务） | 高 | 弱（靠自己实现） | 靠自己拼 | 靠自己实现 |
| **sqlite-vec / sqlite-vss** | 单文件 SQLite 扩展 | **极低** | 十万~百万级 | 借 SQL | 借 SQL 全文 | 借表隔离 |

## 三、结合本项目约束看

项目硬约束（见 charter / 架构）：
- **端侧 1 家 1 实例**，运行环境可能是企业内网/边缘设备，运维能力有限；
- **每企业数据严格隔离**（跨企业泄露是合规红线）；
- 并发以「数十名员工」量级，非高并发；
- 知识源是母婴商品/说明书/FAQ，**典型企业规模约数千~数百万 chunks**，罕见超千万。

据此：

- **Chroma（嵌入式）**：最契合「端侧零运维 + 1家1实例」。单目录即一个企业知识库，天然隔离；
  进程内检索延迟低；开发最快。**风险**：超大规模（>千万向量）或高并发多副本时力不从心，
  且高级过滤/量化能力弱于 Qdrant。
- **Qdrant**：当某客户知识量真正很大、或要更高并发/多副本/强过滤时更稳。它**也可 embedded 模式**
  运行（运维接近 Chroma），所以「先用嵌入式 Qdrant、后续转服务态」是平滑路径。代价：概念与
  部署比 Chroma 重一档。
- **Milvus / 分布式方案**：对本项目**过度设计**，端侧不推荐，除非出现超大规模证据。
- **pgvector**：仅当部署已依赖 Postgres 时顺手；否则为向量库引入 PG 反而增运维。
- **sqlite-vec**：极轻量备选（回退项），仅当边缘资源极端受限、只能跑单文件库时回退；生态与混合检索弱于 Chroma。

## 四、与「企业隔离」的关系（关键）

无论选哪个，**跨企业隔离必须靠「每企业独立命名空间 + 检索时强制 enterprise_id 过滤」**实现，
不能依赖"不同实例自然隔离"就松懈——同一进程内多企业时尤甚。Chroma/Qdrant 都用
**collection per enterprise** + 元数据过滤，足以满足；具体见 `MOD-kb` 验收 `test_kb_isolation`。

## 五、建议（供参考，非决定）

- **默认推荐 Chroma（嵌入式）**：零运维、单目录隔离、覆盖绝大多数母婴企业规模，最快落地。
- **触发切换到 Qdrant 的条件**（写入 charter 作为决策阈值）：单企业知识 > 千万向量、或需高并发
  多副本、或需强 payload 过滤/量化。Qdrant 支持 embedded 起步，切换平滑。
- **混合检索建议默认开启**：母婴「规格参数/成分表」类精确查询，向量 + 关键词融合明显更准。
- embedding 模型单独作为配置项（默认 bge-small-zh / 本地），与向量库解耦。
